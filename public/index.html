Software development vs web development

They're the same! Web development is just a subset of software development.

Programming is writing instructions

You're telling your computer what to do. In this case, we're going to write instructions on how to handle requests from a web browser and return a response.

Programming is creative

It's creative problem solving at its core. Imagine it's like instructions for a LEGO set - it walks through each step, adding a piece or combining different segments until it comes together like the picture on the box.

Our finished product is an application

web app, mobile app, desktop app - they're all software applications with a specific domain. Ruby, the programming language we're going to use, can be used in all three, but we're focusing on building a web app.

Web application vs web page vs website

Different words that mean the same thing. A website is a destination that allows a web application to serve up a web page to the user. Ta da!

Most people refer to static websites as having web pages.

Static vs Dynamic

Static stays the same. Nothing changes unless the code is manually changed. Dynamic changes either based on a response from a database, a 3rd party service, even a clock - a dynamic website is built to accept changes to its content based on specific conditions.

Scaffolds

Creates all the guts you need to create, retrieve, update, and destroy something

Down in the CRUD

Create, Retrieve, Update, and Destroy

Rails creates the model, the related database table, the controller, and the views

Generate your scaffold


  $ rails generate scaffold idea name:string description:text picture:string
After the colon - those are datatypes. The attribute name has a datatype of string. An attribute is a specific piece of a model. A model is a representation of a database object - its name is singular, but you can have many instances of them.

Migrate the database


  $ rake db:migrate
        	

  $ rails server
Rails uses ActiveRecord to talk to the database, so you don't have to. It's like an interpreter that can help you select cultural delicacies while you're abroad. So you can make sure you don't eat sheep face, unless that's your deal.

We're going to add the ability to upload pictures and then save the location of that picture as the string.

Mining for Gems

Gems are Ruby libraries that you can plug in to your application. They live in your Gemfile. Gems add functionality to your web application that saves you time from having to code in yourself. One of the tenants of Rails is DRY - Don't Repeat Yourself. Don't re-invent the wheel if someone else has done the work for you.

Add the Carrierwave gem

Add to Gemfile:


  gem 'carrierwave'

Bundler manages your gems and their integration into your application


  $ bundle

Generate the uploader!


  $ rails generate uploader Picture

Prep the model

In the Idea.rb file (located at app/models/idea.rb) add:


  mount_uploader :picture, PictureUploader

directly underneath the top line.

Prep the form

In app/views/ideas/_form.html.erb, replace


  <%= f.text_field :picture %>

with


  <%= f.file_field :picture %>

Prep the view

In app/views/ideas/show.html.erb, replace


  <%= @idea.picture %>

with


  <%= image_tag(@idea.picture_url, :width => 600) if @idea.picture.present? %>

Fire up the server!


  $ rails server

http://localhost:3000/ideas

But why can't we just go to http://localhost:3000?

Rails follows the routes we define. We have to tell it to respond with the page we want if we want something other than the default

Convention over configuration - one of the central tenets of Rails.

Means a developer only needs to specify unconventional aspects of the application

Let's update the routes

In your text editor, go to


  config/routes.rb

and after the first line add


  root :to => redirect('/ideas')

We'll talk more about routing later.

Restart your server and go to http://localhost:3000

Fantastic!
